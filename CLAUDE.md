## ⚠️ 必須ルール：作業完了後の Gemini 連携ログ出力

**作業（実装・修正・整理・調査）が完了するたびに、必ず以下のログファイルを作成すること。**

### ログの保存先
```
/Users/Daiki/Library/CloudStorage/GoogleDrive-re.accelerator@gmail.com/マイドライブ/ClaudeCode/StrideEdge/Claude_to_Gemini/
```

### ファイル名規則
```
Gemini_report_YYYYMMDD_<作業トピック>.log
例: Gemini_report_20260420_race_repository.log
```

### ログの記載内容（フォーマット）
```
================================================================================
送信者: Daiki（実行者）
日時: YYYY-MM-DD
件名: <作業タイトル>
================================================================================

## 【作業報告：<タイトル>】

**1. 実施内容:** （作成・変更・削除したファイルと概要）
**2. 要件チェック:** （設計思想・AС への適合状況）
**3. 動作確認結果:** （成功 / 失敗 / 未確認）
**4. 次のステップ:** （未実装・残課題）

================================================================================
```

> 同日に同トピックのログが既存の場合は上書きしてよい。

## ⚠️ 必須ルール：変更後のアプリ自動リビルド

**backend・frontend・05_app/start_bg.sh のいずれかを変更した場合、必ず最後に実行すること：**

```bash
bash /Users/Daiki/ClaudeCode/StrideEdge/05_app/build_app.sh
```

このコマンドは Swift バイナリをコンパイルして `.app` を更新し、デスクトップの `StrideEdge.app` を自動で最新版に置き換える。実行を忘れるとデスクトップのアプリが古いままになる。

## 開発チーム体制

```
[PM] 要件定義・スコープ管理
[開発者] 設計・実装（規約違反スクレイピング禁止）
[テスター] 正常系・準正常系・異常系・回帰試験
[レビュアー] 要件漏れ・設計不備・再発防止の確認
```

障害発生時はインシデントチームに切り替え → `01_pm/incident_log.md` に記録。

## 基本方針

- 個人利用前提・有料API不使用
- 法令・利用規約・robots.txt を遵守
- 非公式スクレイピング・ログイン突破は禁止
- ローカル完結を最優先
- 作業後は `logs/` に記録する
- APIキー等の機密情報は `.env` で管理する
